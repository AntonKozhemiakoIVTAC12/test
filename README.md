# CRM Stages — прототип для Joomla 4

Прототип интерфейса и логики стадий CRM: менеджер не может «перепрыгнуть» вперёд без выполнения обязательных действий. Реализован как **компонент Joomla 4** (админка).

## Почему компонент

Выбрана реализация в виде **административного компонента** (`com_crm`), а не модуля или отдельной страницы:

- Есть своя модель данных (компании, события), установка таблиц через манифест.
- Логика стадий и переходов изолирована в сервисе `StageMachine`, без привязки к одному представлению.
- Один пункт меню «CRM» в админке, подменю «Компании» → список и карточка компании.
- В перспективе тот же компонент можно расширить фронтом (карточка для менеджера на сайте) или API.

## Среда и запуск

### Docker (рекомендуется)

```bash
docker compose up -d
```

- Joomla: <http://localhost:8080>
- MySQL: порт 3306, пользователь `joomla`, пароль `joomla`, БД `joomla`.

После первого запуска:

1. Установите Joomla через браузер (localhost:8080).
2. Таблицы CRM создаются при установке компонента. Если компонент подключается копированием файлов (volumes), выполните SQL вручную:
   - Файл `administrator/components/com_crm/sql/install.mysql.utf8.sql` — замените `#__` на префикс таблиц сайта (например `j_`).
3. Добавьте тестовую компанию: выполните `seed.sql`, подставив свой префикс вместо `#__`.
4. В админке: **Components → CRM** (или **Компании**) → список компаний → «Карточка» у нужной компании.

### Локально

Установите Joomla 4 и скопируйте в неё каталоги:

- `administrator/components/com_crm/` → `administrator/components/com_crm/`
- При необходимости установите компонент через **Extensions → Install** (пакет с манифестом и SQL).

## UI: карточка компании

В карточке компании есть:

- **Текущая стадия** — отображается по коду стадии (Ice, Touched, Aware, …).
- **Список доступных действий** — кнопки/формы только для действий, разрешённых на данной стадии и при текущем состоянии (например, «Заполнить дискавери» только после события «разговор с ЛПР»).
- **Блок «Инструкция/скрипт»** — текст для менеджера в зависимости от стадии (что делать дальше).
- **История событий** — хронологический список событий по компании (попытка контакта, разговор с ЛПР, дискавери, демо, счёт, оплата, удостоверение и т.д.).

Действия с дополнительными данными (комментарий, дискавери, дата/время демо) открываются в модальных окнах с формами.

## Логика стадий и событий

### Стадии (порядок)

Ice (C0) → Touched (C1) → Aware (C2) → Interested (W1) → demo_planned (W2) → Demo_done (W3) → Committed (H1) → Customer (H2) → Activated (A1) → Null (N0).

### Ограничения и переходы (Acceptance criteria)

| Код | Стадия       | Нельзя на стадии                          | Вход (доступно если)     | Выход (далее если)           |
|-----|--------------|-------------------------------------------|---------------------------|------------------------------|
| C0  | Ice          | —                                         | —                         | —                            |
| C1  | Touched      | Заявка, КП, планирование/показ демо       | Нет разговора с ЛПР       | Есть разговор с ЛПР          |
| C2  | Aware        | Планирование и показ демо                 | Разговор с ЛПР             | Заполнена форма дискавери    |
| W1  | Interested   | Заявка, КП                                 | Заполнен дискавери        | Заданы дата/время демо       |
| W2  | demo_planned  | Заявка, КП                                 | Есть дата плана демо       | Демо проведено (переход по ссылке) |
| W3  | Demo_done    | —                                         | Демо проведено &lt; 60 дней | Заявка и/или счёт            |
| H1  | Committed    | —                                         | Есть счёт                  | —                            |
| H2  | Customer     | —                                         | Есть оплата                | —                            |
| A1  | Activated    | —                                         | Одно удостоверение         | —                            |

Реализация проверяется unit-тестами в формате Given/When/Then (см. раздел «Тестирование»).

### События (минимум)

- `contact_attempt` — попытка контакта  
- `lpr_conversation` — разговор с ЛПР + комментарий  
- `discovery_filled` — заполнение дискавери  
- `demo_planned` — планирование демо (дата/время)  
- `demo_done` — демо проведено (кнопка/ссылка → событие)  
- `lead_created` — заявка создана  
- `invoice_issued` — счёт выставлен  
- `payment_received` — оплата получена  
- `first_certificate` — первое удостоверение выдано  

Интеграции (телефония, КП, платёжки) допускается мокировать; важно, что логика переходов и ограничений проверяема тестами.

## Архитектура и БД

### Модель данных

- **`#__crm_companies`**  
  - `id`, `name`, `stage_code` (текущая стадия), даты: `discovery_filled_at`, `demo_planned_at`, `demo_done_at`, `invoice_at`, `payment_at`, `first_certificate_at`, `created`, `updated`, `created_by`.  
  - Индексы: `stage_code`, `updated`, `created`.

- **`#__crm_events`**  
  - `id`, `company_id`, `event_type`, `payload` (JSON), `created`, `created_by`.  
  - Индексы: `(company_id, created)`, `event_type`.  
  - Внешний ключ на `#__crm_companies(id)` с ON DELETE CASCADE.

- **`#__crm_stages`**  
  - Справочник: `code`, `title`, `sort_order` (опционально; дублируется в коде в `StageMachine`).

Логика переходов и доступных действий сосредоточена в **`StageMachine`** (без зависимостей от Joomla), что упрощает unit-тесты.

### Масштабирование до ~10k компаний/день

- **Индексация**: составной индекс `(company_id, created)` по событиям для быстрой выборки истории; индекс по `stage_code` и `updated` по компаниям для списков и фильтров.
- **Журнал событий**: все изменения через таблицу событий; при необходимости можно воспроизводить состояние компании по логу (event sourcing).
- **Минимизация блокировок**: запись событий — вставка в `#__crm_events` и обновление одной строки в `#__crm_companies`; без тяжёлых транзакций и блокировок списков.
- **Логирование**: события с `event_type` и `payload` позволяют строить аналитику и аудит без изменения схемы; при росте объёма — партиционирование `#__crm_events` по `created` (по месяцам/годам).
- **Кэш**: список компаний и агрегаты по стадиям можно кэшировать (Joomla cache или Redis); инвалидация при записи события по компании.

## Тестирование

### Перечень unit-тестов (что и зачем)

- **Ограничения по стадиям**: Ice — только «Позвонить»; на Touched без события ЛПР нет «Дискавери»; на Aware нет «Планировать демо» до заполнения дискавери; на Interested после дискавери доступно «Планировать демо»; на Demo_done при демо старше 60 дней недоступны «Заявка»/«КП», при недавнем демо — доступны.
- **Переходы по событиям**: Touched + lpr_conversation → Aware; Aware + discovery_filled → Interested; Interested + demo_planned → demo_planned; demo_planned + demo_done → Demo_done; Demo_done + invoice_issued/lead_created → Committed; Committed + payment_received → Customer; Customer + first_certificate → Activated.
- **Граничные случаи**: Ice + lpr_conversation не меняет стадию (нет перехода «вперёд» без Touched).
- **Вспомогательное**: порядок стадий, названия стадий.

Запуск:

```bash
composer install
vendor/bin/phpunit
```

Лог прогона сохранён в `test-run.log` (в корне проекта).

### Цикл «баг → исправление → зелёные тесты»

- **Ситуация**: тест `test_aware_after_discovery_plan_demo_allowed` падал: ожидалось, что на стадии **Aware** при уже заполненном дискавери действие «Планировать демо» доступно.
- **Причина**: по правилам на стадии **Aware** планирование и показ демо запрещены; после заполнения дискавери компания переходит на стадию **Interested**, и уже там разрешено «Планировать демо».
- **Исправление**: тест переименован в `test_interested_after_discovery_plan_demo_allowed` и переведён на сценарий «стадия Interested, дискавери заполнен, дата демо не назначена» → проверяется наличие `plan_demo` в доступных действиях.
- **Результат**: после правки все 17 тестов проходят (OK).
